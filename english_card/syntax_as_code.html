English Syntax as Code (v2.0)

自然言語を「プログラミングコード」として解釈する構文仕様書

本ドキュメントは、英語の構文を曖昧な自然言語としてではなく、厳密な型定義と関数シグネチャを持つ「コード」として解釈するための公式リファレンスである。「5文型」という伝統的な分類を、ソフトウェアエンジニアリングの視点（型の制約、メモリ操作、実行スレッド）から再構築した。

1. 引数（変数）の定義と語源

英語を構成する各要素は、プログラムにおける変数、インスタンス、メソッド、アノテーションとして厳密に型定義される。

Subject (S) / 主語
語源：sub（下に） + iacere（投げる） ＝ 下に置かれた土台
[プログラミング定義] 処理の実行主体となる Instance または Object。メソッドを呼び出す大元（レシーバー）。
Verb (V) / 動詞
語源：verbum（言葉そのもの）
[プログラミング定義] 主体が実行する Method や関数。文の処理内容（振る舞い）を決定する。状態を更新する代入演算子（=, <-）の役割を果たすこともある。
Object (O) / 目的語
語源：ob（向かって） + iacere（投げる） ＝ 動作の標的
[プログラミング定義] メソッド（Verb）に渡される必須の Argument（引数）。null（欠損値）を許容しない。
Complement (C) / 補語
語源：complere（満たす、補完する）
[プログラミング定義] Subject または Object のプロパティに代入される State や Value。型は文字列（名詞）または論理値・列挙型（形容詞）。
Modifier (M) / 修飾語
語源：modificare（制限する、調整する）
[プログラミング定義] 実行結果のメインロジックに影響を与えない Annotation またはコード内の /* Comment */。型のインターフェース（前置詞＋名詞など）さえ守れば、どこに挿入してもコンパイルエラーにならない。

2. 関数シグネチャ（5つの基本構文）

いわゆる「5文型」は、メソッドの引数の渡し方とメモリ操作のパターン（関数シグネチャ）に過ぎない。Pythonのオブジェクト指向と、Rの関数型アプローチの両面から定義する。

パターン1：自己完結メソッド (SV)

外部引数を必要としない、主体のみで完結するVoid型メソッド。副作用（Side Effect）はSubject内部に限定される。

Python (OOP)
# Syntax: Subject.execute_verb()
def process(sun):
  sun.rise()
R (Functional)
# Syntax: execute_verb(Subject)
process <- function(sun) {
  rise(sun)
}
EXAMPLE: The sun rises. (太陽が昇る)
-> 外部へエネルギー（引数）が漏れず、自己完結する。

パターン2：プロパティの代入 (SVC)

Verbが単なるセッター（代入演算子）として機能し、Subjectの属性（Property）を同期的に更新する。

Python
# Syntax: Subject.state = Complement
soup.state = "hot"
R
# Syntax: Subject$state <- Complement
soup$state <- "hot"
EXAMPLE: The soup is hot. (そのスープは熱い)
-> The soup = hot の関係。Verbは繋ぐだけの連結動詞。

パターン3：引数実行メソッド (SVO)

外部の引数（Object）を受け取り、それに対して処理を実行する単一引数関数。

Python
# Syntax: Subject.execute_verb(Object)
i.read(book)
R
# Syntax: execute_verb(Subject, Object)
read(subject = i, object = book)
EXAMPLE: I read a book. (私は本を読む)
-> I が book に対して read を実行。

パターン4：複数引数のルーティング (SVOO)

IndirectObject（宛先）と DirectObject（ペイロード）の2つの引数を転送する。

Python
# Syntax: Subject.execute_verb(target, payload)
he.give(target=me, payload=pen)
R
# Syntax: execute_verb(Subj, target, payload)
give(he, target = me, payload = pen)
EXAMPLE: He gave me a pen. (彼は私にペンをくれた)
-> 宛先 me に 実データ a pen を転送。

パターン5：オブジェクト状態の書き換え (SVOC)

高階関数/副作用の極致。Subjectが処理を実行した副作用として、文の中に「小さな代入式（O=C）」がスレッドとして発生する。

Python
# Nested Logic (Side Effect)
they.make(him)
him.state = "happy" # 同期的な状態変化
R
# Mutable state simulation
him <- make(they, target = him)
attr(him, "state") <- "happy"
EXAMPLE: They made him happy. (彼らは彼を幸せにした)
-> make 処理の結果、him = happy という状態が動的に生成される。

3. ランタイム実行：Stockfish Engine の状態空間（12の時制と相）

英語の時制（Tense）と相（Aspect）は、チェスエンジン（Stockfish）のGUI画面（コンソール）における **「評価値の表示状態」** のメタファーとして直感的に理解できる。

  • Past (過去): PGNに書き出された過去ログ。画面にはもう表示されていない（グレーアウト）。
  • Present (現在): 今まさに計算中で、画面にアクティブに表示されている（実色）。
  • Future (未来): キューに積まれたタスク。これから画面に表示される予定のもの（点線枠）。

1. Present (Root Node / 現在の盤面)

今まさにGUI画面にアクティブに表示されている実色の評価値。

🔴 Present Simple (現在単純形)

The engine evaluates the position.

【状態: Point / 点】
静的評価（Static Evaluation）。時間幅を持たず、特定のノードにおける瞬間的なスナップショット値（スコア）。

// 0次元的な静的評価値
const score = evaluate(current_pos);

↗️ Present Progressive (現在進行形)

The engine is searching the lines.

【状態: Vector / ベクトル・波】
現在進行形でCPUリソースを消費し、探索を実行中。状態が動的に遷移しているベクトル。

// 1次元的なリソースの流動
while (time_allocated > 0) {
  search_tree(current_pos);
  yield; // 計算を継続
}

🟩 Present Perfect (現在完了形)

The engine has cached the position.

【状態: Area / 面・領域】
過去の探索結果がTT（Transposition Table）に保存され、現在のノード評価を「面」としてカバーしている完了状態。

// 2次元的なカバー範囲（キャッシュ）
if (TT.contains(hash)) {
  return TT.get(hash).score;
}

🟦 Present Perf. Prog. (現在完了進行形)

The engine has been calculating depth 40.

【状態: Volume / 体積・蓄積】
過去から継続して同一局面の探索を行っており、到達深度（Depth）やノード数が蓄積されている状態。

// 3次元的な計算量の積み上げ
while (engine.uptime() > 0) {
  depth.accumulate();
  update_stats(depth);
}

2. Past (Game History / 過去のPGN履歴)

すでにPGN等にログとして保存され、現在のGUI画面からは消えたグレーアウト状態。

⚪ Past Simple (過去単純形)

The engine played e4.

【状態: Past Point】
履歴に書き込まれた単なる点（Ply）。現在の探索スレッドには影響を与えない完了したトランザクション。

// 履歴への固定化
game_history.push({
  move: "e2e4",
  time: "yesterday"
});

〰️ Past Progressive (過去進行形)

The engine was analyzing the blunder.

【状態: Past Vector】
過去の特定の時間枠において、エンジンがリソースを割いて探索を行っていたという状態ログ。

// 過去の特定時点での状態トレース
let log = logs.get("move_15");
assert(log.status === "searching");

🔲 Past Perfect (過去完了形)

The engine had already found a mate.

【状態: Past Area】
過去の特定時点を基準とした時、それより以前の探索によって既に結論の面が構築され、準備されていた状態。

// 過去のある時点でのキャッシュヒット
let snapshot = db.get("ply_20");
assert(snapshot.mate_found === true);

📦 Past Perf. Prog. (過去完了進行形)

The engine had been pondering the move.

【状態: Past Volume】
過去のある時点までずっと計算量を蓄積し続けていたという計算資源の塊の記録。

// 過去の蓄積エネルギー量
let stats = db.get("opponent_turn");
assert(stats.nodes > 1_000_000);

3. Future (Future Nodes / 探索木の未来展開)

これから計算され、未来のタイミングでGUI画面に表示される予定の点線枠（予測）。

⭕ Future Simple (未来単純形)

The engine will evaluate this branch.

【状態: Future Point】
探索キューに積まれたタスク（Promise）。将来、到達した瞬間に静的評価（点）が行われる予定。

// タスクキューへの追加
task_queue.push(node);
// or
Promise.resolve(evaluate(node));

〽️ Future Progressive (未来進行形)

The engine will be searching the endgame.

【状態: Future Vector】
探索が特定の手順まで進んだ未来時点で、そこで探索スレッドがアクティブになっているだろうという予測。

// 未来のフェーズ移行予測
let phase = estimate(depth + 10);
expect(phase).toBe("endgame_search");

🏁 Future Perfect (未来完了形)

The engine will have solved the puzzle.

【状態: Future Area】
未来の特定時点において、必要な評価空間が完全にカバー（完了・面化）し終わっているという状態の確約。

// 未来の完了条件の充足
await search_until(depth == 30);
state.is_solved = true;

📶 Future Perf. Prog. (未来完了進行形)

The engine will have been running for hours.

【状態: Future Volume】
未来の特定時点において、累積したノード数が巨大な体積となり、依然として増え続けている状態予測。

// 未来時点での累積リソース予測
let time = target_time - start_time;
assert(time > hours(5));

4. SVOOの引数設計：ターゲットとペイロード

パターン4（SVOO）において、2つの引数（Object）は等価ではない。「宛先情報」と「実データ」という役割の違いがある。

  • IndirectObject (O1): Routing Target（宛先情報 / オプション）
  • DirectObject (O2): Payload（送信される実データ / 必須）

原則としてPayloadが必須であり、Targetは省略するか、Modifierとして外部に追い出すことが可能である。

Python: Parameter Downgrade
# 1. Full Parameters (SVOO)
subject.send(target=him, payload=gift)

# 2. Downgrade to Modifier (SVO + M)
# targetがアノテーション化
subject.send(payload=gift) # [Mod: to him]

# 3. Target Omission (SVO)
subject.send(payload=gift)
R: Parameter Downgrade
# 1. Full Parameters (SVOO)
send(target = him, payload = gift)

# 2. Downgrade to Modifier (SVO + M)
# Mは attributes 等で付加されるイメージ
res <- send(payload = gift)
attr(res, "target") <- "him"

# 3. Target Omission (SVO)
send(payload = gift)

5. Modifier（修飾語）の実践的デバッグ

実際の英文が複雑化し、SMVMOM のように肥大化する原因は、すべて Modifier（M）の追加によるものである。構文解析を行う際は、これらをコメントアウトしてメインロジックから切り離すことで、全体の構造が明確になる。

Python: Parsed Syntax
# [Modifier: Yesterday]
subject_I \
  .execute_bought( # [Mod: who usually save...]
    object_a_book  # [Mod: about space]
  )
# [Modifier: safely]
# [Modifier: on the internet]

# Execution Logic:
# subject_I.execute_bought(object_a_book)
R: Parsed Syntax
# attr: Yesterday
bought(
  subject = I, # attr: who usually save...
  object = a_book # attr: about space
)
# attr: safely
# attr: on the internet

# Execution Logic:
# bought(subject = I, object = a_book)
Original Text:
Yesterday, I, who usually save money, bought a book about space safely on the internet.

Parsed (Mを不可視化):
Yesterday, I, who usually save money, bought a book about space safely on the internet.

英語読解における最大のコツは、前置詞（in, at, for, to等）や関係代名詞（who, which等）を見つけた瞬間に、そこから先を /* ... */ として脳内で不可視化することである。
