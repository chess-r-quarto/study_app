English Syntax as Code

自然言語を「プログラミングコード」として解釈する構文仕様書（完全版）

本ドキュメントは、英語の構文を曖昧な自然言語としてではなく、厳密な型定義と関数シグネチャを持つ「コード」として解釈するための公式リファレンスである。「5文型」という伝統的な分類を、ソフトウェアエンジニアリングの視点から再構築した。

1. 引数（変数）の定義と語源

英語を構成する各要素は、プログラムにおける変数、インスタンス、メソッド、アノテーションとして定義される。

Subject（主語）

語源：sub（下に） + iacere（投げる） ＝ 下に置かれた土台

[プログラミング定義] 処理の実行主体となる Instance または Object。メソッドを呼び出す大元。

Verb（動詞）

語源：verbum（言葉そのもの）

[プログラミング定義] 主体（Subject）が実行する Method や関数。文の処理内容（振る舞い）を決定する。状態を更新する代入演算子（=）の役割を果たすこともある。

Object（目的語）

語源：ob（向かって） + iacere（投げる） ＝ 動作の標的

[プログラミング定義] メソッド（Verb）に渡される必須の Argument（引数）。nullを許容しない。

Complement（補語）

語源：complere（満たす、補完する）

[プログラミング定義] Subject または Object のプロパティに代入される State や Value。型は文字列（名詞）または論理値・列挙型（形容詞）。これがないとメソッドの意味が未完成になる。

Modifier（修飾語）

語源：modificare（制限する、調整する）

[プログラミング定義] 実行結果のメインロジックに影響を与えない Annotation またはコード内の /* Comment */。型のインターフェース（前置詞＋名詞、副詞など）さえ守れば、どこに挿入してもコンパイルエラー（文法エラー）にならない。

2. 関数シグネチャ（5つの基本構文）

いわゆる「5文型」は、メソッドの引数の渡し方（関数シグネチャ）のパターンに過ぎない。

パターン1：自己完結メソッド (SV)

外部引数を必要としない、主体のみで完結するVoid型メソッド。

// Syntax: Subject & Verb
Subject.executeVerb();
Example
The sun rises.
太陽が昇る。
-> 外部へエネルギー（引数）が漏れず、自己完結する（自動詞）。

パターン2：プロパティの代入 (SVC)

Verbが単なるセッター（代入演算子 =）として機能し、Subjectの状態を更新する。

// Syntax: Subject & Verb = Complement
Subject.property = Complement;
Example
The soup is hot.
そのスープは熱い。
-> The soup = hot の関係。Verbは繋ぐだけの連結動詞（Linking Verb）。

パターン3：引数実行メソッド (SVO)

外部の引数（Object）を受け取り、それに対して処理を実行する。

// Syntax: Subject & Verb to Object
Subject.executeVerb(Object);
Example
I read a book.
私は本を読む。
-> Iがbookに対してreadを実行。I ≠ book。

パターン4：複数引数のルーティング (SVOO)

IndirectObject（受取人・アドレス）と DirectObject（対象物・ペイロード）の2つの引数を順番に渡す。

// Syntax: Subject & Verb to IndirectObject to DirectObject
Subject.executeVerb({
  target: IndirectObject,
  payload: DirectObject
});
Example
He gave me a pen.
彼は私にペンをくれた。
-> target: me に payload: a pen を転送。

パターン5：オブジェクト状態の書き換え (SVOC)

Subjectが処理を実行した結果（副作用）として、ObjectのプロパティがComplementに書き換わる。文の中に「小さな文（O=C）」が入れ子構造（スモール・クローズ）として含まれる。

// Syntax: Subject & Verb to (Object = Complement)
Subject.executeVerb(Object);
Object.state = Complement; // 副作用による状態変化
Example
They made him happy.
彼らは彼を幸せにした。
-> make処理の結果、him = happy という状態が生成される。

3. SVOOの引数設計：ターゲットとペイロード

パターン4（SVOO）において、2つの引数（Object）は等価ではない。プログラミングの視点から見ると、「宛先」と「実データ」という役割の違いがある。

  • IndirectObject (O1 / 人): Routing Target（宛先情報）
  • DirectObject (O2 / 物): Payload（送信される実データ / 直接の対象）

原則：Payload（O2）が必須であり、Target（O1）はオプション

動詞が直接働きかけるのは O2（物）である。そのため、原則として O1（人）は省略するか、Modifier（修飾語）として外部に追い出す（to/for をつける）ことが可能である。「SVOOはMの派生」と感じる直感の正体はこれである。

// 1. フルパラメーター (SVOO)
Subject.send(target_Him, payload_Gift);

// 2. Mへのダウングレード (SVO + M) -> targetを外部アノテーション化
Subject.send(payload_Gift) /* [Modifier: to Him] */;

// 3. Targetの省略 (SVO) -> 宛先が自明、または重要でない場合
Subject.send(payload_Gift);
End-Focus（文末重心）の法則
英語のメモリ配置ルールとして、「配列（文）の後ろにある変数ほど、新しく重要なデータとして処理される」という原則がある。
・send(him, a gift) → Payloadである gift に焦点。
・send(a gift, to him) → 宛先である him に焦点。

例外：Payload（O2）が消える特殊なメソッド

原則とは逆に、「O1（宛先・人）だけを残し、O2（実データ・物）を省略できる」オーバーロードされたメソッド（動詞）が存在する。これは、処理の主目的が「データを送ること」から「対象（人）に影響を与えること」にシフトした場合に起こる。

Exception: targetを直接操作するメソッド
She teaches kids. (SVO)
彼女は子供たちを教えている。
-> 本来は teaches(kids, math) のようにペイロードが必要だが、「子供たちを指導する」という対象操作自体が目的となっているため、O2が省略（Null許容）されている。

4. Modifier（修飾語）の実践的デバッグ

実際の英文が複雑化し、SMVMOM のように肥大化する原因は、すべて Modifier（M）の追加によるものである。構文解析（リーディング）を行う際は、これらをコメントアウトしてメインロジックから切り離すことで、全体の構造が明確になる。

/* [Modifier: Yesterday] */
Subject_I /* [Modifier: who usually save money] */
  .executeVerb_bought(
    Object_a_book /* [Modifier: about space] */
  )
/* [Modifier: safely] */
/* [Modifier: on the internet] */;
Example (Parsed)
Yesterday, I, who usually save money, bought a book about space safely on the internet.
昨日、普段はお金を貯めている私が、宇宙に関する本をインターネットで安全に買った。
-> /* M */ を全て無視すると、実行されるメインロジックは I bought a book (Subject & Verb to Object) のみとなる。

英語読解における最大のコツは、前置詞（in, at, for, to等）や関係代名詞（who, which等）を見つけた瞬間に、そこから先を /* ... */ として脳内で不可視化することである。
